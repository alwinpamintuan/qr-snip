# Contributing to QR Snip

QR Snip is developed feature by feature with privacy, predictable browser behavior, and evidence-backed compatibility as release constraints. Contributions should keep the default workflow simple: explicitly activate, select one QR code, review the result, then choose an action.

Participation in project spaces is governed by the [Code of Conduct](CODE_OF_CONDUCT.md). Usage questions and non-security reports should follow [SUPPORT.md](SUPPORT.md); suspected vulnerabilities must use the [private security process](.github/SECURITY.md).

## Development setup

Requirements:

- Node.js 22.13 or newer
- pnpm 9 or newer
- A current Chromium-based browser and Firefox 140+ for cross-browser changes

Install from the committed lockfile:

```bash
pnpm install --frozen-lockfile
```

Run Chromium or Firefox development mode:

```bash
pnpm dev
pnpm dev:firefox
```

WXT writes generated state to `.wxt/` and browser packages to `.output/`. These directories are build products and should not be treated as source.

## Before making a change

1. Read the relevant product requirement in [docs/PRODUCT_SPEC.md](docs/PRODUCT_SPEC.md).
2. Read [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) for the runtime boundary you will touch.
3. Review [docs/SECURITY.md](docs/SECURITY.md) for changes involving decoded values, URLs, browser permissions, messaging, the clipboard, persistence, or network access.
4. Check [docs/IMPLEMENTATION_PLAN.md](docs/IMPLEMENTATION_PLAN.md) and [docs/ROADMAP.md](docs/ROADMAP.md) before introducing a new feature or abstraction.

If a change deliberately alters a documented product or security decision, update the documentation in the same commit.

Notable user-visible changes should also update the `[Unreleased]` section of [CHANGELOG.md](CHANGELOG.md).

## Architecture and code standards

Follow SOLID and DRY principles at the repository's actual boundaries:

- Keep privileged WebExtension APIs in `entrypoints/` or narrowly scoped adapters.
- Keep application sequencing in `src/application/`.
- Keep geometry, classification, message guards, and security assessment pure where possible.
- Keep `src/ui/` responsible for presentation and gestures, not navigation policy.
- Run CPU-intensive pixel work in the decoder worker, not on the page's main thread.
- Add new decoders or result interpreters behind focused interfaces rather than expanding a controller with format-specific branches.
- Share security decisions through one implementation and validate them again at the privileged boundary.
- Prefer precise domain names, discriminated unions, explicit return types, and short single-purpose functions.
- Use comments for browser constraints, security invariants, and non-obvious tradeoffs—not to narrate syntax.
- Avoid speculative frameworks or abstractions. A new dependency needs a concrete capability, bundle impact, maintenance assessment, and security review.

The host page, screenshot pixels, runtime messages, and decoded payload must all be treated as untrusted input.

## Testing changes

During development, run the narrowest relevant test in watch mode. `pnpm check` is the fast local gate for types, deterministic unit tests, and locales. Decoder changes should also run `pnpm test:corpus`. Before submitting, run the complete gate:

```bash
pnpm check:full
```

The full gate adds Chromium and Firefox production builds, generated-manifest assertions, Firefox package validation, and browser automation.

Add tests with the change:

- Geometry and coordinate behavior belongs in `tests/selection.test.ts`.
- Payload classification and protocol behavior belongs in `tests/result.test.ts`.
- Destination warnings belong in `tests/link-security.test.ts`.
- Runtime message shapes belong in `tests/messages.test.ts`.
- RGBA resource constraints belong in `tests/decode-pipeline.test.ts`.
- Decoder compatibility belongs in `tests/fixture-corpus.test.ts` and the QR fixture manifest.

Tests must not print screenshots, full image buffers, real credentials, or sensitive payloads. Synthetic examples should use reserved domains such as `example.com`.

## QR fixture workflow

The deterministic corpus is generated by `scripts/generate-qr-fixtures.mjs` and described by `tests/fixtures/qr/manifest.json`.

Regenerate it with:

```bash
pnpm fixtures:generate
pnpm test
```

Fixture changes must include:

- expected payload and decode expectation;
- version, error-correction level, quiet-zone, and transformation metadata where applicable;
- deterministic generation inputs or a documented source;
- redistributable licensing information;
- positive and negative coverage that protects against false positives.

Review the generated diff before committing. Do not add real-world screenshots that may contain personal, copyrighted, or confidential information.

Compatibility claims in the README must be supported by maintained fixtures. Put experimental designs under **Best effort** until the corpus demonstrates the documented release threshold.

## UI contributions

The overlay follows a Material 3 Expressive visual language without loading a component framework or remote font into arbitrary pages.

- Preserve the closed Shadow Root and system-font-only implementation.
- Use existing CSS custom properties and semantic colors before adding literals.
- Keep controls at least 44 × 44 CSS pixels.
- Maintain visible focus, readable hierarchy, and operability at 200% text scaling.
- Test light, dark, `prefers-contrast: more`, and `prefers-reduced-motion` modes.
- Motion should explain state changes and must not delay input or obscure the decoded value.
- Render all untrusted text with `textContent`; never inject payloads through `innerHTML`.

Do not hide or visually de-emphasize a destination hostname in favor of a friendlier label.

## Security and permission changes

Changes to permissions, content security policy, external messaging, web-accessible resources, storage, history, network access, or executable formats require an explicit security review.

The following behaviors are prohibited unless the product and threat model are intentionally redesigned:

- automatic navigation after decoding;
- remote screenshot or payload processing;
- remote URL metadata, redirect, reputation, product, or tracking lookup by default;
- broad persistent host permissions used for convenience;
- rendering decoded HTML, SVG, Markdown, scripts, or data URLs as active content;
- persisting screenshots or scan history by default.

If permissions change intentionally, update `scripts/assert-manifest.mjs`, the README, architecture, security model, QA plan, and store disclosures together.

## Browser testing

For changes that touch capture, injection, workers, messaging, clipboard behavior, CSS, or browser APIs:

1. Run `pnpm build:all`.
2. Load `.output/chrome-mv3/` in at least one Chromium browser.
3. Load `.output/firefox-mv3/manifest.json` through Firefox temporary add-on debugging.
4. Exercise success, no-result, suspicious-link, Copy, Open, retry, and Escape behavior.
5. Verify restricted pages fail with a recoverable toolbar message.
6. Record browser versions and any untested platform in the pull request.

Use only synthetic fixture pages for screenshots or bug reports.

## Documentation style

- Describe current behavior in present tense.
- Put speculative work in the implementation plan or post-1.0 roadmap.
- Distinguish **tested**, **best effort**, and **unsupported** compatibility.
- Include exact commands, file paths, assumptions, and acceptance criteria for developer tasks.
- Keep product, architecture, security, tests, and generated-manifest documentation consistent.

## Commit and pull-request checklist

Prefer small, focused commits with an imperative summary, for example `docs: clarify Brave installation` or `fix: reject control characters in URLs`.

Before requesting review, confirm:

- [ ] The behavior and scope are documented.
- [ ] New logic has focused tests.
- [ ] `pnpm check:full` passes.
- [ ] Chromium and Firefox behavior was tested or the gap is stated.
- [ ] Permissions and generated manifests remain intentional.
- [ ] No screenshot, payload, URL, or sensitive data is logged or persisted.
- [ ] Accessibility and reduced-motion behavior were considered for UI changes.
- [ ] Compatibility claims are backed by fixtures.
- [ ] The change does not introduce automatic opening or remote lookup.
